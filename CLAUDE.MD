## Deploy

Editar `puntoflix_firebase.html` dispara automáticamente dos hooks en `~/.claude/settings.json`:
1. Copia y despliega a Firebase (async)
2. `git add` + `git commit` + `git push` a GitHub (async)

Si falla el hook de Firebase:
```powershell
Copy-Item "C:\Users\jozz1\Downloads\PROYECTOS CLAUDE\GESTOR DE USUARIOS\puntoflix_firebase.html" C:\puntoflix\public\index.html -Force
Set-Location C:\puntoflix; firebase deploy --only hosting
```

Si falla el hook de GitHub:
```powershell
Set-Location "C:\Users\jozz1\Downloads\PROYECTOS CLAUDE\GESTOR DE USUARIOS"
git add puntoflix_firebase.html; git commit -m "manual sync"; git push
```

El repo también versiona `index.html` (redirect a `puntoflix_firebase.html`), `CNAME` (app.puntoflix.com), `sw.js` (service worker network-first, copia de `C:\puntoflix\public\sw.js`) y `404.html`.

**Unir clientes duplicados ya existentes (`detectarClientesDuplicados`/`abrirUnirDuplicados`/`renderUnirDupMod`/`confirmarUnirDuplicados`, estado `S.dupMod`):** herramienta de limpieza en **Ajustes → Datos → "🔗 Clientes Duplicados"** (muestra el nº detectado). `detectarClientesDuplicados()` agrupa: (1) por **teléfono** (≥7 dígitos iguales) → seguro; (2) por **nombre** `norm()` entre los que no cayeron en (1), y **descarta** el grupo si hay 2+ teléfonos distintos entre los miembros (probables homónimos). Devuelve `[{tipo:'tel'|'nombre', principal, otros[]}]`; **principal** = el de más suscripciones (desempata: con teléfono, luego el primero). Los grupos viven en la variable de módulo **`_dupGroups`** (no se persiste); selección en `S.dupSel` (Set de índices, todos por defecto). El modal lista cada grupo (los "nombre" con badge ⚠️ "revisa"). `confirmarUnirDuplicados` mueve las subs de cada `otro` al `principal` **clonándolas** (`{...s}`, para no compartir referencia con la copia de papelera), completa teléfono/notas, reactiva, **reasigna los pagos** (`p.clientId`→principal) y manda el duplicado vacío a **`S.papelera`** (con sus subs originales + `_unidoA`, recuperable, auto-purga 30 días). Enganchado en `renderApp`, `closeAll` y el guard `editando` del sync (para que un sync no reemplace `S.clients` con el modal abierto y deje `_dupGroups` colgando). Solución para la base YA duplicada; el `submitCMod` evita crear nuevos.

**Anti-duplicados al crear cliente (`submitCMod`, modo `add`):** antes de crear, se busca un cliente EXISTENTE y, si lo hay, se **ANEXA la nueva suscripción a ese cliente** en vez de duplicar (variable local `mergeTarget`). Detección: (1) **teléfono** normalizado a dígitos, ≥7 dígitos iguales → match fuerte, anexa **sin preguntar**; (2) si no, **nombre** vía `norm()` (sin tildes/mayúsculas: "Solórzano"=="solorzano") → match débil, pide `confirm` (Aceptar=anexar / Cancelar=crear aparte, por si es homónimo). Al anexar: `newClientId=mergeTarget.id` (para que el pago inicial y el modal de bienvenida apunten al cliente existente), `mergeTarget.subs=mergeTarget.subs.concat(subs)`, completa `phone` si faltaba, reactiva si estaba Suspendido/Cancelado, y concatena `notes`. El modal de bienvenida muestra "➕ Suscripción Añadida" (flag `S.welcomeMerged`, reseteado en `cerrarWelcome`) y el toast dice "Suscripción añadida a X (sin duplicar)". Esto es la solución de raíz al problema de registros duplicados de la misma persona; complementa la agrupación por teléfono en recordatorios. La **importación por Excel (`confirmImport`)** usa la misma lógica: agrupa las filas por `ph:digits`(≥7) o `nm:norm(nombre)`, y contra los clientes existentes une por teléfono (si ambos tienen, deben coincidir) o por nombre normalizado — así no duplica al importar. Los **Combos** (`submitCombo`) todavía NO deduplican (tienen su propio modo 'existing' manual). [[prefer-grouping-over-editing-data]]

**Anti-pérdida de guardado (varias capas, no quitar):** (1) `_svCore` **devuelve un estado** `'ok'|'offline'|'error'|'sin-sesion'` y `sv()` lo propaga; los flujos críticos (p.ej. `submitCMod`) muestran un toast HONESTO según ese estado en vez de un "✅" fijo (antes mentía aunque el guardado fallara). (2) **Sesión ausente:** si en el momento de guardar `auth.currentUser` es null (refresco de token), `_svCore` ya NO aborta en silencio: respalda en localStorage bajo `_ultimoUid` y **reintenta** ~4 veces (1.2 s c/u) a que vuelva la sesión. `_guardarEmergencia(uid)` centraliza el snapshot local (antes inline). (3) **Online con transacción fallida:** se hace `ref.set` y se **espera** el resultado real (no fire-and-forget) para no reportar éxito falso; offline sí queda fire-and-forget + `'offline'`. (4) **Rescate fino al cargar** (`ld`): si la nube cargó pero el respaldo local es **fresco (≤2 h)** y tiene items por id que la nube perdió (clients/cuentas/pagos/proveedores), se re-inyectan y se re-sube con `sv()` (no revive lo que esté en papelera; ventana de 2 h para no resucitar borrados de otro dispositivo). **(4b) Bandera "pendiente de subir" (jul 2026 — fix del caso real "creé un cliente en la mañana y no se guardó"):** `_marcarPendiente(uid)`/`_limpiarPendiente(uid)` manejan `localStorage 'pf_pendiente_<uid>'`. Se MARCA en `_svCore` justo tras `_guardarEmergencia` (también en el branch sin-sesión, con `_ultimoUid`) **y además SINCRÓNICAMENTE en el propio `sv()` ANTES de encolar `_svCore`** (jul 2026): si `_svCore` queda detrás de otro guardado en vuelo en `_svCola` y la app se cierra en ese instante, el respaldo local + la bandera ya se escribieron (el cambio no se pierde aunque `_svCore` nunca llegue a correr). `sv()` usa `auth.currentUser?.uid||_ultimoUid`; `_svCore` los repite (idempotente). Se LIMPIA SOLO cuando la nube confirmó, en los 4 desenlaces exitosos (transacción ok, set forzado tras fusión, `.then` del set offline, set directo online). Si al abrir la app la bandera sigue puesta, el rescate fino de `ld()` corre **SIN el límite de 2 h** (`fresco = ≤2h || pendiente`) — la bandera prueba que el respaldo contiene un cambio que nunca llegó a la nube; sigue respetando las papeleras. Si con bandera no hay nada que rescatar (el cambio llegó por la cola offline del SDK), se limpia ahí mismo. **(4c) Recuperar RENOVACIONES no subidas (jul 2026 — fix del caso real "cobré con 💰 un toque y al recargar/al otro día volvió a pendiente"):** el rescate por-id de (4)/(4b) solo AÑADE items nuevos; pero una renovación **MODIFICA el `endDate` de una cuenta que YA existe** en la nube (💰 `cobreRapido`/`_cobrarPares`, 🛒 modales, masiva, 💸 `pagueRapido`), así que si ese guardado no llegaba a la nube, al recargar la nube tenía la fecha vieja y el rescate lo ignoraba → el PAGO sí se recuperaba (id nuevo) pero la fecha volvía atrás ("pagó pero sigue vencido"). Fix: en `ld()`, **solo con la bandera `pendiente` puesta**, se recorre cada sub/cuenta que existe en ambos y, si el respaldo local tiene una fecha (`endDate`/`renewalDate`) **más nueva (posterior)** que la nube, se aplica junto con `startDate`/`pagado`. Es **MONÓTONO**: solo AVANZA la fecha, nunca la retrocede, así **jamás pisa una renovación más reciente hecha en otro dispositivo** (si la nube es más nueva, gana la nube). Suma a `rescatados` → re-sube con `sv()`. Comparación de fechas como string `'YYYY-MM-DD'` (orden cronológico). No cubre ediciones arbitrarias de campos (precio/nombre) sin base 3-way — solo el avance de fecha de renovación, que es seguro por construcción. (5) **`beforeunload`** avisa si `_syncEstado` es `'guardando'`/`'error'` (no cerrar con un cambio sin subir). (6) **Banner rojo fijo** (`#syncBanner`, z-index 100000, `_mostrarBannerSync`/`_ocultarBannerSync`/`_evaluarBannerSync` llamados desde `setSync`): aviso fuerte que el puntito discreto no da. **Control de concurrencia (multi-dispositivo):** el documento se guarda **completo** (no por campos), así que sin protección "el último que escribe gana" y un dispositivo/pestaña con datos viejos puede pisar un cambio nuevo de otro (causa real de "lo cambié y se revirtió", p. ej. una transferencia). Fix: el doc lleva un campo `rev` (entero incremental); `aplicarDatosFirestore` registra `S._rev` al cargar/sincronizar; `sv()` escribe dentro de un `db.runTransaction` que lee la `rev` del servidor y, si avanzó respecto a `S._rev` (otro dispositivo guardó), **no pisa**: lanza conflicto. **En conflicto NO se descarta el cambio local** (antes sí → se perdían clientes recién creados): `_fusionarConflicto(serverData)` guarda el estado local, aplica lo del servidor (deja `S._rev` correcto) y **fusiona por id** las listas (`clients`, `pagos`, `cuentas`, `proveedores`, `papelera*`). **Fusión 3-WAY (jul 2026):** existe una **base** `_syncBase` (mapa por colección `id → JSON` del ÚLTIMO estado del servidor visto por este dispositivo; `_syncBaseFrom(d)` la reconstruye al final de `aplicarDatosFirestore` y tras CADA guardado exitoso en los 4 desenlaces de `_svCore` — transacción ok, set forzado, set offline, set directo). En la fusión, un item local **idéntico a la base no se tocó aquí → gana el SERVIDOR** (no se pisa ni se revive si otro dispositivo lo borró); solo ganan los items locales **cambiados o nuevos**. Antes lo local ganaba SIEMPRE y un dispositivo con datos viejos (p.ej. la PC con un snapshot saltado por los guards) **revertía renovaciones hechas en el celular** al guardar cualquier cosa (marcar "enviado", etc.) — el pago quedaba pero la fecha volvía a la vieja ("pagó pero sigue vencido"). Además `S.waEnviados`/`S.waSeguimiento` se fusionan por **unión** (server + local encima) para no perder el marcado que disparó el guardado. Los borrados de otro dispositivo se respetan filtrando lo que esté en la papelera. Luego `_svCore` **reintenta** el guardado (hasta 3 veces, parámetro `_intentos`); si tras 3 intentos sigue habiendo conflicto, fuerza un `ref.set` con lo fusionado (no perder el cambio). `_svCore` recibe `_intentos` y `sv()` lo encadena igual. Si la transacción falla por estar **sin conexión**, cae al `ref.set(payload)` directo (se encola offline) para no romper el modo offline; ahí sí puede haber clobber, pero offline es de un solo dispositivo. Tras escribir, `S._rev=payload.rev`. Docs legacy sin `rev` se tratan como `rev=0` y migran solos en el primer guardado. **Serialización:** `sv()` es un wrapper que encadena cada guardado al anterior (`_svCola=_svCola.then(_svCore,_svCore)`) y el cuerpo real está en `_svCore()`; así nunca corren dos guardados a la vez en el mismo dispositivo (evita que un guardado automático en 2.º plano y uno manual se pisen o disparen un falso conflicto). Todas las escrituras a Firestore pasan por aquí; no escribir al doc directo desde otro lado.
